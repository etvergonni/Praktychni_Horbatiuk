## 1. Опис роботи

Розширено систему управління навчальною групою з Практичної №1 додатковим функціоналом симуляції лабораторного обладнання через порти вводу/виводу. Створено клас `Port` (один порт з 64-байтним буфером), клас `PortMatrix` (двовимірний масив 16×16 портів), клас `PortLogger` (логування через `StringBuilder`). Клас `Student` доповнено одновимірним масивом оцінок за лабораторні роботи `byte[] LabGrades`. Реалізовано інтерфейс `ICloneable` для класів `Port` та `Student`. Меню розширено до 15 пунктів.

## 2. Архітектура

- **`Enums.cs`** — переліки `StudentStatus` та `DeviceType`.
- **`PortExceptions.cs`** — власні винятки: `PortException`, `PortClosedException`, `PortMatrixIndexException`.
- **`Student.cs`** — клас студента з масивом `LabGrades[10]`, прив'язкою до порту, `ICloneable`.
- **`GradeJournal.cs`** — журнал оцінок за предметами (з ПР №1, переписаний на `StringBuilder`).
- **`Port.cs`** — один порт із буфером, `ICloneable`.
- **`PortMatrix.cs`** — двовимірний масив `Port[,]` 16×16 + методи Open/Close/Read/Write/Scan/Find.
- **`PortLogger.cs`** — логування через накопичувальний `StringBuilder`.
- **`StudentGroup.cs`** — управління групою + інтеграція з матрицею портів (прив'язка студентів, симуляція лабораторних).
- **`Program.cs`** — точка входу з меню на 15 пунктів.

## 4. Відповіді на контрольні питання

### 4.1 Різниця між одновимірними та двовимірними масивами

**Одновимірний масив** — це лінійна послідовність елементів, доступ до яких здійснюється за одним індексом. У моїй роботі прикладами є:
- `byte[] LabGrades` у класі `Student` — зберігає 10 оцінок за лабораторні роботи (доступ: `LabGrades[i]`);
- `byte[] DataBuffer` у класі `Port` — 64-байтний буфер передачі даних.

**Двовимірний масив** — це таблиця з рядками й колонками, де елементи адресуються двома індексами. Приклад у моїй роботі:
- `Port[,] _matrix` у класі `PortMatrix` — таблиця 16×16 портів (доступ: `_matrix[row, col]`).

Принципова різниця: одновимірний масив підходить для лінійних послідовностей, а двовимірний — для табличних структур, де природньо існують поняття «рядок» і «колонка». У моєму випадку матриця 16×16 наочно моделює реальну лабораторну стійку, де порти розташовані рядами та колонками.

### 4.2 Чому StringBuilder ефективніший за конкатенацію

Рядки в C# **незмінні** (immutable). Тому при кожній операції конкатенації (`+` або `+=`) створюється новий об'єкт рядка, а попередній копіюється у нього. Якщо склеювати 1000 рядків через `+`, програма виконає приблизно 1000 операцій виділення пам'яті та 1000 копіювань.

`StringBuilder` тримає внутрішній буфер змінного розміру, який автоматично розширюється у міру потреби. Усі операції `Append` модифікують **той самий** буфер, без зайвих копіювань. Це особливо помітно при роботі з великими обсягами даних.

Приклад з мого класу `PortLogger`:

```csharp
// ПОГАНО — створює новий рядок на кожне додавання
string log = "";
for (int i = 0; i < 1000; i++)
    log += $"[{DateTime.Now}] подія #{i}\n";  // 1000 копіювань!

// ДОБРЕ — те ж саме, але одним буфером
var sb = new StringBuilder();
for (int i = 0; i < 1000; i++)
    sb.Append('[').Append(DateTime.Now).Append("] подія #").Append(i).AppendLine();
```

У методі `GenerateBigReport()` моєї роботи генерується звіт зі 100+ записів. На такому обсязі різниця у швидкості між `StringBuilder` і конкатенацією може сягати десятків разів.

### 4.3 Як реалізована симуляція портів

Симуляція побудована на двох рівнях:

**Рівень 1 — один порт** (клас `Port`). Кожен порт має:
- унікальний `PortNumber` (від 0 до 255 у моїй матриці);
- одновимірний масив `byte[] DataBuffer` розміром 64 байти — імітує буфер передачі;
- прапорець `IsOpen` (відкритий / закритий);
- назву пристрою `DeviceName` та тип `DeviceType`;
- методи `Open()`, `Close()`, `Write(byte[] data)`, `Read()`, `ClearBuffer()`.

Запис даних можливий лише в **відкритий** порт — при спробі запису в закритий кидається власний виняток `PortClosedException`.

**Рівень 2 — матриця** (клас `PortMatrix`). Двовимірний масив `Port[,] _matrix = new Port[16, 16]` зберігає 256 портів. Доступ за координатами `(row, col)` з перевіркою меж — при виході за межі кидається `PortMatrixIndexException`. Метод `ScanMatrix()` обходить усю матрицю подвійним циклом і повертає список відкритих портів.

Використання масивів — критичне:
- `byte[]` для буфера — дозволяє швидко передавати довільні бінарні дані;
- `Port[,]` для матриці — природньо моделює фізичне розташування портів.

### 4.4 Як інтегровано масиви в систему управління студентами

Інтеграція реалізована на трьох рівнях:

1. **У класі `Student`** додано поле `byte[] LabGrades` (10 елементів — кожен індекс відповідає номеру лабораторної). Доступ через метод `AddLabGrade(int labNumber, byte grade)`, який перевіряє діапазон номера (1–10) та значення оцінки (0–100). Метод `GetAverageLabGrade()` обчислює середнє за ненульовими елементами масиву.

2. **У класі `Student`** також додано координати прив'язки до порту: `PortRow` та `PortCol`. Якщо обидва ≥ 0 — студент має «лабораторне місце».

3. **У класі `StudentGroup`** додано метод `AssignStudentToPort(Student, row, col)`, який встановлює координати та перевіряє, чи це місце ще не зайняте. Метод `SimulateLab(...)` поєднує все разом: записує оцінку в `LabGrades`, відкриває порт, записує контекстні дані (номер лабораторної, оцінку, дату) у буфер порту через `Write()`, і додає запис у `PortLogger`.

### 4.5 Логіка роботи з матрицею 16×16

Матриця ініціалізується методом `Initialize()`: подвійний цикл `for (int r = 0; r < 16; r++)` зовнішньо по рядках і `for (int c = 0; c < 16; c++)` усередині по колонках створює 256 портів. Тип пристрою визначається за номером рядка: рядок 0 — осцилографи, 1 — мультиметри, 2 — генератори сигналів, 3 — блоки живлення, 4 — логічні аналізатори, далі шаблон повторюється. Кожен порт отримує унікальний номер за формулою `r * 16 + c`.

Доступ до конкретного порту здійснюється через `GetPort(row, col)`, який перед поверненням перевіряє координати через `ValidateCoordinates()`. Якщо `row` або `col` виходить за межі 0–15, кидається `PortMatrixIndexException`.

Сканування матриці (`ScanMatrix()`) повертає список усіх **відкритих** портів. Пошук за типом пристрою (`FindByDeviceType()`) повертає всі порти певного типу — корисно, наприклад, щоб знайти всі осцилографи.

Виведення стану матриці (`GetMatrixView()`) формується через `StringBuilder` і показує наочну таблицю 16×16, де кожна клітинка позначена `.` (закритий) або `O` (відкритий).

### 4.6 Труднощі та їх вирішення

- **Інтеграція з кодом ПР №1.** Старий клас `Student` не містив масиву оцінок за лабораторні та координат порту. Довелося акуратно розширити клас, не зламавши існуючу логіку (журнал предметів, валідація, ICloneable).
- **Реалізація `ICloneable` для `Student`.** Проста копія через `MemberwiseClone()` не підходила, бо в полі `LabGrades` зберігається посилання на масив — копія мала б змінюватись синхронно з оригіналом. Рішення: виконати глибоку копію масиву через `(byte[])LabGrades.Clone()` та окремо скопіювати журнал оцінок.
- **Серіалізація масиву `LabGrades` у JSON.** Стандартна десеріалізація не могла відновити поле, бо в класі є `required`-властивість. Довелося використати ручний розбір через `JsonDocument` (як і в ПР №1) та окремо відновити елементи масиву по індексах.
- **Обробка винятків.** Спочатку всі помилки об'єднувались під `Exception`. Це не давало користувачу зрозумілого повідомлення. Створила власну ієрархію `PortException → PortClosedException, PortMatrixIndexException` і додала окремі блоки `catch` у меню — тепер для кожного типу помилки виводиться зрозумілий текст.

### 4.7 Порівняння масивів з колекціями

**Масиви** — це структури **фіксованого розміру** з прямим доступом за індексом. Швидкі, ефективні за пам'яттю, але неможливо динамічно змінювати розмір.

**`List<T>`** — це динамічна обгортка над масивом, яка автоматично розширюється при потребі. Зручніша для роботи, але має невеликий overhead на додаткові поля та виклики методів.

У моїй роботі я **свідомо** обрала масиви у трьох місцях:
- **`byte[] LabGrades` (10 елементів)** — кількість лабораторних робіт у курсі **фіксована** (10 у семестрі). Динамічний `List<byte>` тут зайвий. Крім того, фіксований індекс дозволяє звертатись «оцінка за лабораторну №5» напряму — `LabGrades[4]`.
- **`byte[] DataBuffer[64]` у класі `Port`** — буфер фізичного порту має фіксований розмір згідно з апаратною специфікацією. Використання `List<byte>` не відповідало б реальності.
- **`Port[,] _matrix[16, 16]`** — матриця також фіксована. У C# **двовимірних `List<>` як вбудованого типу взагалі немає** — тільки `List<List<T>>`, що складніше у використанні та менш ефективне.

У всіх інших місцях (список студентів у групі, перелік знайдених портів) використано `List<T>`, бо розмір там динамічний.

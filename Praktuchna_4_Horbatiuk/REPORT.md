## 1. Що таке перевантаження операторів. Які можна, а які не можна перевантажувати

Перевантаження операторів - це можливість задати власну поведінку стандартних знаків (+, -, >, ==, тощо) для своїх класів. Завдяки цьому з об'єктами можна працювати так само природно, як з числами: наприклад, `v1 + v2` для векторів або `student1 > student2` для студентів.

Можна перевантажувати: арифметичні (+, -, *, /, %), унарні (++, --, +, -, !), порівняння (==, !=, >, <, >=, <=), оператори true/false та оператори приведення типу (implicit/explicit).

Не можна перевантажувати: оператор присвоєння (=), оператор доступу до члена (.), умовні && та || напряму (вони працюють через true/false і & та |), оператор індексації [] напряму (його роль виконує індексатор), тернарний (?:), а також new, sizeof, typeof.

## 2. Приклади перевантажених операторів з коду (не менше 6)

1. `Vector operator +(Vector a, Vector b)` - додавання векторів покоординатно.
2. `Vector operator *(Vector v, double scalar)` - скалярне множення вектора на число.
3. `explicit operator double(Vector v)` - приведення вектора до double (повертає довжину).
4. `bool operator >(Student a, Student b)` - порівняння студентів за середнім балом.
5. `Student operator +(Student a, Student b)` - об'єднання двох студентів у командний профіль.
6. `bool operator true(GradePoint g)` - оцінка вважається достатньою, якщо >= 8.
7. `implicit operator GradePoint(double v)` - неявне перетворення числа в оцінку.
8. `Fraction operator +(Fraction a, Fraction b)` - додавання дробів зі скороченням.
9. `StudentGroup operator +(StudentGroup a, StudentGroup b)` - об'єднання двох груп.

## 3. Різниця між явним та неявним приведенням типів. Де використано

**Неявне (implicit)** приведення виконується автоматично, без явного запису, і застосовується коли перетворення безпечне і не втрачає дані. Використано в класі `GradePoint`: `GradePoint <-> double` в обидва боки, тому можна писати `GradePoint g = 7.5;` без явного приведення.

**Явне (explicit)** приведення вимагає, щоб програміст сам написав `(тип)`, і застосовується коли перетворення може бути несподіваним або втрачати інформацію. Використано в класі `Vector`: `(double)v` повертає довжину вектора. Оскільки вектор це не просто число, перетворення зроблено явним, щоб програміст усвідомлено його викликав.

## 4. Для чого потрібні оператори true та false. Приклад

Оператори true та false дозволяють використовувати об'єкт безпосередньо в логічній умові (if, while, тернарний оператор), ніби це bool. У класі `GradePoint` оператор true повертає істину, якщо оцінка >= 8.

Приклад з коду:
```
GradePoint g = 9.0;
if (g)
    Console.WriteLine("Достатня оцінка");
```
Тут `if (g)` спрацьовує саме завдяки оператору true: оскільки 9 >= 8, умова істинна.

## 5. Як реалізовано порівняння студентів. Які оператори

У класі `Student` перевантажено оператори >, <, >=, <=, ==, != . Вони порівнюють студентів за середнім балом (а оператор == враховує ще й рівність прогресу навчання). Завдяки цьому пошук найкращого студента у методі `BestStudent()` написано природно:
```
if (s > best) best = s;
```
Оскільки перевантажено ==, додатково перевизначено методи `Equals` та `GetHashCode`, щоб поведінка була узгодженою.

## 6. Труднощі та як вирішувалися

- Оператори порівняння треба перевантажувати парами: якщо є `>`, обов'язково має бути `<`; якщо `>=`, то й `<=`. Інакше код не компілюється. Вирішено реалізацією повних пар.
- При перевантаженні `==` компілятор вимагає також перевизначити `Equals` і `GetHashCode`. Додано обидва методи.
- Для дробів (Варіант 2) треба було коректно скорочувати результат. Вирішено через обчислення найбільшого спільного дільника (алгоритм Евкліда) у конструкторі, тому 2/4 одразу стає 1/2.

## 7. Як перевантаження операторів покращує читабельність коду

Замість громіздких викликів методів код читається як звичайна математика. Порівняйте:
- Без операторів: `vectorAdd(v1, v2)`, `studentCompare(s1, s2) > 0`.
- З операторами: `v1 + v2`, `s1 > s2`.

Другий варіант коротший, зрозуміліший і ближчий до того, як ми звикли працювати з числами. Це зменшує кількість помилок і робить наміри коду очевидними.

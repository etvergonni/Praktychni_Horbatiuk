## Завдання для звіту

### 1. Що таке наслідування? Які переваги та недоліки цього механізму?

Наслідування - це механізм, який дозволяє створювати новий клас (похідний) на основі вже наявного (базового), переймаючи його властивості та методи. Похідний клас отримує доступ до членів базового класу і може додавати власні або змінювати успадковану поведінку.

У роботі базовим класом є UniversityMember, від нього успадковується Person, далі Student, а від Student похідні класи ExcellentStudent, ForeignStudent, WorkingStudent, GraduateStudent. Кожен наступний рівень додає власну специфіку, не дублюючи код попереднього.

Переваги: повторне використання коду (спільна логіка пишеться один раз у базовому класі), легше розширення системи (новий тип студента додається без зміни наявних класів), єдиний інтерфейс для роботи з різними типами через базовий клас, що відкриває шлях до поліморфізму.

Недоліки: сильний зв'язок між базовим і похідними класами (зміна базового може зачепити всі похідні), занадто глибока ієрархія ускладнює розуміння коду, не всі відношення між класами правильно описуються наслідуванням (іноді доречніша композиція).

### 2. Наведіть приклади використання base у вашому коді (конструктори та методи)

Виклик конструктора базового класу. Клас Student у своєму конструкторі передає ПІБ і дату народження конструктору Person:

```
public Student(string fullName, DateTime dateOfBirth, string recordBookNumber)
    : base(fullName, dateOfBirth)
{
    RecordBookNumber = recordBookNumber;
}
```

Похідні класи від Student через base передають дані вже конструктору Student:

```
public ExcellentStudent(string fullName, DateTime dateOfBirth, string recordBookNumber, string achievement)
    : base(fullName, dateOfBirth, recordBookNumber)
{
    Achievement = achievement;
}
```

Виклик методу базового класу. Похідні класи доповнюють інформацію базового методу через base.GetInfo():

```
public override string GetInfo()
{
    return base.GetInfo() + $" [відмінник: {Achievement}]";
}
```

### 3. Поясніть різницю між virtual / override та new (приховуванням методу)

virtual і override працюють у парі та реалізують справжнє перевизначення (поліморфізм). Метод базового класу позначається virtual, а похідний перевизначає його через override. При виклику через посилання на базовий тип виконується саме та версія методу, яка відповідає реальному типу об'єкта.

new (приховування методу) лише ховає метод базового класу новою реалізацією. При виклику через посилання на базовий тип виконається версія базового класу, а не похідного. Тобто поведінка залежить від типу посилання, а не від реального об'єкта.

У роботі скрізь використано саме virtual / override (GetInfo, CalculateScholarship, Enroll), бо потрібен поліморфізм: коли група перебирає List<UniversityMember> і викликає m.GetInfo(), має спрацювати метод реального типу (наприклад, ForeignStudent), а не базового.

### 4. Для чого потрібні абстрактні класи? Де ви їх використали?

Абстрактний клас - це клас, від якого не можна створити об'єкт напряму, він слугує лише основою для похідних. Абстрактний клас може містити абстрактні методи (без тіла), які зобов'язані реалізувати всі похідні класи. Це задає спільний контракт для ієрархії.

У роботі два абстрактні класи. UniversityMember - вершина ієрархії з абстрактним методом CalculateScholarship(): кожен тип члена університету зобов'язаний визначити, як рахується його стипендія. Person - проміжний абстрактний клас: сам по собі об'єкт Person не має сенсу (потрібен конкретний тип, студент), але він збирає спільні властивості ПІБ, дата народження, пошта.

### 5. Навіщо використовують sealed класи та методи?

Ключове слово sealed забороняє подальше наслідування. Запечатаний клас не може бути базовим для інших класів, а запечатаний метод не можна перевизначити в похідних класах. Це використовують, коли клас завершений за змістом і його розширення не передбачене, а також для захисту логіки від випадкових змін і для невеликого приросту продуктивності.

У роботі sealed застосовано до класу GraduateStudent (аспірант). Це найвищий, завершальний рівень студентської ієрархії, від нього не передбачено похідних класів, тому він запечатаний.

### 6. Як ви реалізували поліморфізм у системі студентів?

Поліморфізм реалізовано через роботу зі спільним базовим типом. Колекція групи має тип List<UniversityMember>, тобто зберігає членів університету будь-якого конкретного типу. Методи працюють із цією колекцією через базовий тип, а конкретна поведінка визначається під час виконання:

GetMembersInfo() перебирає всіх членів і викликає m.GetInfo(): для кожного об'єкта спрацьовує версія методу його реального типу (Student, ExcellentStudent, ForeignStudent тощо).

GetTotalScholarship() підсумовує m.CalculateScholarship() по всіх членах: кожен тип рахує стипендію за власною формулою, хоча виклик однаковий.

GetMembersByType<T>() - узагальнений (generic) метод, що повертає членів конкретного типу через фільтрацію OfType<T>().

Таким чином, один і той самий виклик дає різний результат залежно від реального типу об'єкта, що і є суттю поліморфізму.

### 7. Які труднощі виникали при проектуванні ієрархії класів? Як їх вирішували?

Основна трудність виникла з рівнем оголошення методу GetInfo(). Спершу він був оголошений virtual у класі Person. Але коли група працює зі списком через базовий тип UniversityMember і викликає GetInfo(), компілятор не знаходив метод, бо на рівні UniversityMember його не було. Рішення: підняти GetInfo() на рівень UniversityMember (оголосити там virtual), а в Person зробити його override. Після цього поліморфний виклик через базовий тип запрацював.

Друга трудність - правильно розподілити, що належить базовому класу, а що похідному, щоб не дублювати код. Спільні дані (ПІБ, дата, пошта) винесено в Person, навчальні дані (заліковка, середній бал) в Student, а специфіку (досягнення, країна, місце роботи, тема дослідження) у відповідні похідні класи.

### 8. Що нового ви дізналися під час виконання цієї практичної роботи?

Навчилася будувати багаторівневу ієрархію класів і правильно передавати дані вгору ланцюжком через base. Зрозуміла практичну різницю між abstract, virtual, override і sealed, а головне, навіщо кожне з них на реальному прикладі. Побачила, як абстрактний клас задає контракт для всіх похідних через абстрактний метод. Найважливіше, на практиці відчула суть поліморфізму: робота зі списком базового типу, коли один виклик методу дає різну поведінку залежно від реального об'єкта.
